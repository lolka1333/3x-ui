# Анализ максимального количества клиентов в одном inbound

## Резюме

После изучения кодовой базы проекта 3x-ui, **не обнаружено явных программных ограничений** на максимальное количество клиентов в одном inbound. Однако существуют практические ограничения, связанные с архитектурой системы.

## Технические детали

### 1. Хранение данных

- **Поле Settings**: Клиенты хранятся в поле `Settings` модели `Inbound` как JSON массив
- **Тип данных**: `string` в Go, что соответствует типу `TEXT` в SQLite
- **Ограничение SQLite TEXT**: До 2^31-1 байт (≈2 ГБ) на поле

```go
// database/model/model.go:52
Settings string `json:"settings" form:"settings"`
```

### 2. Структура хранения

Клиенты хранятся в JSON формате внутри поля Settings:
```json
{
  "clients": [
    {
      "id": "...",
      "email": "...",
      "password": "...",
      ...
    },
    ...
  ]
}
```

### 3. Обработка клиентов в коде

В коде нет проверок на максимальное количество клиентов:

- `AddInboundClient()` - просто добавляет клиентов в массив без проверки размера
- `GetClients()` - извлекает всех клиентов из JSON без ограничений
- Все операции работают с массивом клиентов напрямую

```go
// web/service/inbound.go:597-661
func (s *InboundService) AddInboundClient(data *model.Inbound) (bool, error) {
    // ... нет проверки на максимальное количество
    oldClients := oldSettings["clients"].([]any)
    oldClients = append(oldClients, interfaceClients...)
    oldSettings["clients"] = oldClients
    // ...
}
```

## Практические ограничения

### 1. Размер JSON

Приблизительный расчет:
- Один клиент в JSON: ~200-500 байт (в зависимости от полей)
- Теоретический максимум: ~4-10 миллионов клиентов (при размере поля 2 ГБ)

### 2. Производительность

**Проблемы при большом количестве клиентов:**

1. **Парсинг JSON**: При каждом обращении к inbound необходимо парсить весь JSON массив клиентов
   ```go
   // web/service/inbound.go:130-141
   func (s *InboundService) GetClients(inbound *model.Inbound) ([]model.Client, error) {
       settings := map[string][]model.Client{}
       json.Unmarshal([]byte(inbound.Settings), &settings) // Парсинг всего массива
       // ...
   }
   ```

2. **Обновление конфигурации Xray**: При изменении клиентов необходимо перезагрузить конфигурацию Xray, что может быть медленным при большом количестве клиентов

3. **Поиск клиентов**: Операции поиска клиента по email выполняются линейно по всему массиву
   ```go
   // web/service/inbound.go:168-186
   func (s *InboundService) checkEmailsExistForClients(clients []model.Client) (string, error) {
       // Линейный поиск по всем клиентам
       for _, client := range clients {
           // ...
       }
   }
   ```

4. **Синхронизация с Xray API**: При каждом изменении все клиенты передаются в Xray

### 3. Ограничения Xray-core

Xray-core сам по себе не имеет жестких ограничений на количество клиентов в одном inbound, но:
- Большое количество клиентов может замедлить обработку соединений
- Увеличивается потребление памяти
- Может замедлиться проверка аутентификации клиентов

## Рекомендации

### Практические лимиты:

1. **Для нормальной производительности**: 1000-5000 клиентов на inbound
2. **Максимально допустимо**: 10000-50000 клиентов (с возможными проблемами производительности)
3. **Не рекомендуется**: Более 50000 клиентов в одном inbound

### Альтернативные подходы:

1. **Разделение на несколько inbounds**: Вместо одного большого inbound использовать несколько меньших
2. **Использование отдельных портов**: Разные inbounds на разных портах для лучшей изоляции
3. **Оптимизация структуры данных**: Для очень больших объемов можно рассмотреть вынос клиентов в отдельную таблицу

## Выводы

**Технически**: Максимальное количество клиентов ограничено только размером поля TEXT в SQLite (≈2 ГБ), что теоретически позволяет хранить миллионы клиентов.

**Практически**: Рекомендуется не более 1000-5000 клиентов на один inbound для обеспечения нормальной производительности системы.

**В коде**: Нет явных ограничений или валидации максимального количества клиентов при добавлении.
